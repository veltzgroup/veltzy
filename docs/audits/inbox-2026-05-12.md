## Achados por Dimensao

### Dimensao 1: FUNCIONAL

#### F1. 🔴 Critico -- `pipeline_id` ausente na RPC `get_conversation_list`
**Arquivo:** `supabase/migrations/030_fix_conversation_list_rpc.sql`
**Problema:** O `ChatHeader` usa `lead.pipeline_id` para mostrar nome do pipeline, mas a RPC nao retorna `pipeline_id`. Campo `undefined` → `pipelines.find()` retorna `undefined` → nada exibido. Erro silencioso.
**Acao:** Adicionar `l.pipeline_id` no SELECT e no RETURNS TABLE da RPC.

#### F2. 🔴 Critico -- `first_response_at` ausente na RPC
**Arquivo:** `supabase/migrations/030_fix_conversation_list_rpc.sql`
**Problema:** A RPC nao retorna `first_response_at`. O campo e populado pelo service/edge function, mas o mapeamento em `messages.service.ts:86` tenta ler de `row.first_response_at` que vem `undefined` → sempre `null` no frontend.
**Acao:** Adicionar `l.first_response_at` no SELECT e no RETURNS TABLE da RPC.

#### F3. 🟠 Alto -- Filtro `waiting_internal` nao esta no dropdown de status
**Arquivo:** `src/components/inbox/conversation-list.tsx:38`
**Problema:** O enum de status inclui `waiting_internal`, existe dot color no `conversation-item.tsx`, mas o `<Select>` do filtro nao tem opcao para filtra-lo. Leads nesse status so aparecem no "Todos".
**Acao:** Adicionar `<SelectItem value="waiting_internal">Aguardando interno</SelectItem>`.

#### F4. 🟠 Alto -- Sem som de notificacao para novas mensagens
**Arquivo:** `src/hooks/use-messages.ts`
**Problema:** O spec da Fase 3 exige "Sound alert when message arrives in background". O realtime subscription recebe a mensagem mas nao emite nenhum som. O usuario nao sabe que recebeu mensagem se nao estiver olhando.
**Acao:** Tocar audio notification no INSERT quando `sender_type === 'lead'` e a aba nao esta em foco ou o leadId selecionado e diferente.

#### F5. 🟡 Medio -- `markAsRead` so atualiza `conversation_status`, nao atualiza `is_read` nas mensagens
**Arquivo:** `src/services/messages.service.ts:49-55`
**Problema:** `markAsRead` faz UPDATE em `leads.conversation_status = 'read'` mas NAO atualiza `messages.is_read = true` para as mensagens individuais. O `unread_count` da RPC conta `is_read = false AND sender_type = 'lead'` → o badge de unread continua aparecendo mesmo apos abrir a conversa. O badge so some no proximo polling (15s).
**Acao:** Adicionar UPDATE em `messages SET is_read = true WHERE lead_id = X AND is_read = false AND sender_type = 'lead'`.

#### F6. 🟡 Medio -- Sem paginacao/virtualização na MessageList
**Arquivo:** `src/components/inbox/message-list.tsx`
**Problema:** Todas as mensagens sao renderizadas no DOM. Conversas longas (200+ mensagens) degradam performance. Sem lazy loading ou scroll virtual.
**Acao:** Implementar `react-virtuoso` ou paginacao cursor-based.

#### F7. 🟢 Baixo -- Template de resposta substitui o conteudo ao inves de inserir
**Arquivo:** `src/components/inbox/chat-input.tsx:100`
**Problema:** `onSelect={(t) => { setContent(t) }}` sobrescreve o que o usuario ja digitou. Se o vendedor digitou algo e decide complementar com template, perde o texto.
**Acao:** Mudar para `setContent(prev => prev ? prev + '\n' + t : t)`.

### Dimensao 2: DADOS

#### D1. 🟡 Medio -- `last_customer_message_at` nunca e populado pelo service do frontend
**Arquivo:** `src/services/messages.service.ts`
**Problema:** `sendMessage()` e `routeMessage()` nao atualizam `last_customer_message_at` quando a mensagem e de `sender_type = 'lead'`. Esse campo provavelmente so e populado via webhook. Se a mensagem manual vier de lead (cenario test/seed), o SLA nao funciona.
**Acao:** Verificar se o webhook popula corretamente. Se nao, adicionar trigger no DB.

#### D2. 🟡 Medio -- RPC retorna leads SEM mensagens
**Arquivo:** `supabase/migrations/030_fix_conversation_list_rpc.sql:103`
**Problema:** O WHERE filtra apenas `company_id`. Leads que nunca tiveram mensagem aparecem na lista de conversas com "Sem mensagens". Isso polui o inbox com leads que sao apenas de pipeline.
**Acao:** Adicionar filtro `AND EXISTS (SELECT 1 FROM veltzy.messages m WHERE m.lead_id = l.id)` ou `AND lm.content IS NOT NULL`.

#### D3. 🟢 Baixo -- Casting manual excessivo no mapeamento da RPC
**Arquivo:** `src/services/messages.service.ts:62-109`
**Problema:** O retorno da RPC e mapeado manualmente com `as string`, `as number`, etc. Fragil e dificil de manter. Se um campo muda na RPC, o erro e silencioso.
**Acao:** Criar tipo TypeScript espelhando o RETURNS TABLE da RPC e usar Supabase typed client.

### Dimensao 3: INTEGRACAO

#### I1. 🔴 Critico -- `zapi-send` NAO valida resposta da API Z-API
**Arquivo:** `supabase/functions/zapi-send/index.ts:74-80`
**Problema:** `await provider.sendMessage(config, {...})` e chamado sem checar o retorno. Se a Z-API retorna erro (numero invalido, sessao expirada, rate limit), a mensagem e inserida no DB como `source: 'whatsapp'` mesmo sem ter sido entregue. O vendedor pensa que enviou.
**Acao:** Capturar retorno do `provider.sendMessage()`, checar sucesso. Se falhar: ou nao inserir a mensagem, ou inserir com flag de erro.

#### I2. 🟠 Alto -- Mensagens de media via WhatsApp usam `getPublicUrl()` que pode nao ser acessivel externamente
**Arquivo:** `src/components/inbox/chat-input.tsx:68-69`
**Problema:** Upload usa `getPublicUrl()` do Supabase Storage. Se o bucket `chat-attachments` nao esta publico, a URL nao funciona para a Z-API baixar a media. Se esta publico, qualquer pessoa com a URL acessa o arquivo (seguranca).
**Acao:** Usar `createSignedUrl()` com TTL adequado para envio via Z-API, e manter bucket privado.

#### I3. 🟡 Medio -- Lista de conversas usa polling (15s) ao inves de Realtime
**Arquivo:** `src/hooks/use-conversation-list.ts:21`
**Problema:** Novas mensagens demoram ate 15s para aparecer na sidebar. Em demo, isso e perceptivel: cliente manda mensagem no WhatsApp, vendedor nao ve imediatamente na lista.
**Acao:** Adicionar subscription Realtime na tabela `messages` para invalidar query `['conversations']` instantaneamente.

#### I4. 🟢 Baixo -- Instagram send nao envia tipos de midia, apenas texto
**Arquivo:** `src/services/messages.service.ts:169`
**Problema:** O body enviado para `instagram-send` so inclui `leadId, content, companyId`. Nao envia `messageType`, `fileUrl`, etc. Imagens/audio nao sao roteados para Instagram.
**Acao:** Incluir campos de midia no payload do invoke.

### Dimensao 4: UX/VISUAL

#### U1. 🔴 Critico -- Inbox nao e responsivo (mobile quebrado)
**Arquivo:** `src/pages/inbox.tsx:46`
**Problema:** Layout fixo `w-[340px]` + `flex-1`. Em telas < 640px, ambos paineis aparecem espremidos, inutilizavel em mobile. O sidebar nao esconde nem vira full-width.
**Acao:** Implementar layout mobile: em telas pequenas, mostrar so a lista OU so o chat, com botao de voltar. Padrao: `hidden md:block` no sidebar quando conversa selecionada.

#### U2. 🟠 Alto -- Botao "Ver no Pipeline" navega sem contexto do lead
**Arquivo:** `src/components/inbox/chat-header.tsx:59`
**Problema:** `navigate('/pipeline')` vai para a pagina do pipeline sem passar nenhum identificador do lead. O usuario nao sabe onde encontrar o lead no kanban.
**Acao:** Navegar com query param: `navigate(`/pipeline?lead=${lead.id}&pipeline=${lead.pipeline_id}`)` e fazer o pipeline abrir o card desse lead.

#### U3. 🟠 Alto -- ReplyTemplatesPopover fecha com clique fora nao tratado
**Arquivo:** `src/components/inbox/reply-templates-popover.tsx`
**Problema:** O popover e uma `div` absoluta controlada por state. Nao tem `onClickOutside` — so fecha pelo botao "Fechar" ou ao selecionar template. Se o usuario clica fora, o popover fica flutuando.
**Acao:** Usar `useRef` + `useEffect` com event listener para fechar ao clicar fora, ou migrar para Radix Popover.

#### U4. 🟡 Medio -- Sem indicador de fonte/canal na conversa
**Arquivo:** `src/components/inbox/conversation-item.tsx`
**Problema:** Nao ha icone indicando se a conversa e WhatsApp, Instagram, ou manual. O vendedor nao sabe por qual canal responder ate abrir o chat.
**Acao:** Adicionar icone do canal (WhatsApp, Instagram, etc.) baseado no `lead_sources.slug`.

#### U5. 🟡 Medio -- AudioBubble sempre mostra "Transcricao indisponivel"
**Arquivo:** `src/components/inbox/audio-bubble.tsx:107-109`
**Problema:** Nao existe funcionalidade de transcricao implementada. A mensagem "Transcricao indisponivel" aparece em TODO audio, poluindo visualmente sem agregar valor.
**Acao:** Remover o fallback "Transcricao indisponivel" ate ter a feature implementada. Mostrar so quando `transcription` existir.

---

### Dimensao 5: COMERCIAL

#### C1. 🟠 Alto -- Sem metricas de SLA visiveis para o gestor
**Problema:** O `sla_breached` aparece como badge vermelho no ConversationItem, mas nao ha dashboard/resumo de quantos leads estao em breach, tempo medio de resposta, first response time. O gestor nao tem visibilidade da performance da equipe no inbox.
**Acao:** Adicionar metricas de inbox no dashboard: leads em breach, tempo medio de resposta, FRT medio.

#### C2. 🟡 Medio -- Nao ha fluxo de "transferir conversa"
**Problema:** Vendedor nao consegue transferir uma conversa para outro vendedor ou para um gestor. Se precisa de ajuda, nao tem como escalar dentro do inbox. Tem que ir ao pipeline e reatribuir manualmente.
**Acao:** Adicionar opcao "Transferir para..." no menu do ChatHeader.

#### C3. 🟡 Medio -- Reply templates sem CRUD acessivel
**Problema:** Existe service para create/update/delete templates, mas nao ha UI para gerencia-los. O admin nao consegue criar ou editar templates de resposta sem acesso direto ao banco.
**Acao:** Adicionar tela de gerenciamento de templates em Settings.

## Conclusao

O Inbox tem base solida: realtime, multi-canal, optimistic updates, SLA, audio. Porem, **22 gaps** foram encontrados, sendo **3 criticos** (RPC incompleta, falta de validacao Z-API, mobile quebrado) e **6 altos**. O Sprint 1 resolve os bloqueadores de demo e pode ser feito em 1 dia de trabalho focado.
